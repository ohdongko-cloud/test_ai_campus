# /prd-review — PRD 리뷰 (품질 게이트)

PRD가 구현하기에 충분한지 **루브릭 8항목**으로 점검한다. 기준은 `CLAUDE.md`(§6·특화9)와 8섹션 템플릿.

## 절차
1. 대상 PRD를 읽는다(인자 경로 → 없으면 `docs/prd/` 최근 문서).
2. 아래 8항목을 각각 **✅ 통과 / ⚠️ 보강 / ❌ 미흡**으로 판정한다.
3. ⚠️·❌ 항목마다 **구체적 수정 문장**을 제시한다(추상적 지적 금지).
4. 원하면 PRD를 직접 보강 편집한다.

## 루브릭 8항목
| # | 항목 | 합격 기준 |
|---|---|---|
| 1 | 성공 기준 검증가능 | 클릭·동작으로 확인되는 체크박스 + "빌드 통과" 포함 |
| 2 | 비목표 명시 | 스코프 제외 경계가 적혀 크리프를 막는가 |
| 3 | 엣지 케이스 실질 | 실패·경계·동시성 입력이 표로 정리됐는가 |
| 4 | 보안·PII 영향 | §6 정책에 매핑됐는가(인증·권한·레이트리밋·PII·에러통일·enumeration) |
| 5 | DB 마이그레이션 | 신규 컬럼/테이블·멱등·하위호환·기본값이 명시됐는가 |
| 6 | 모바일·안드로이드 | `app/m/*` 라우트·`versionCode` 영향이 고려됐는가 |
| 7 | 의존성·사전조건 | env·외부서비스·선행 PRD가 적혔는가 |
| 8 | 미해결 질문 | 결정 못 한 항목을 숨기지 않고 드러냈는가 |

> 4·5·6은 해당 기능이 그 영역을 **건드리지 않으면** "해당 없음(N/A)"으로 표기 가능. 단, 판단 근거를 한 줄 남긴다.

## 출력 형식
```
[prd-review] docs/prd/2026-06-06-foo.md

1. 성공기준 검증가능 ✅
2. 비목표 명시 ⚠️ — "관리자 일괄 편집"이 목표/비목표 어디에도 없음 → 비목표에 명시 권장
3. 엣지 케이스 ❌ — 동시 좋아요 중복 클릭 케이스 누락 → 표에 "중복 클릭 | 1회만 반영" 추가
...
8. 미해결 질문 ✅

종합: ⚠️ 보강 후 구현 권장 (❌ 1, ⚠️ 2)
다음: 보강 → /prd-commit (구현 커밋)
```
- ❌가 있으면 "구현 전 보강 필요", 전부 ✅/N/A면 "구현 착수 가능".

## 참고

- `/prd-new`로 만든 PRD를 이어서 점검하는 흐름.
