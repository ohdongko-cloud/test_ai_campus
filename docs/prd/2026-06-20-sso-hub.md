# PRD: SSO 허브(IdP) — AI캠퍼스를 사내 중앙 로그인 허브로 승격

- 작성일: 2026-06-20
- 작성자: Claude (요청자: <오너>)
- 범위: AI캠퍼스를 사내 Next.js 앱들의 중앙 IdP로 만드는 OIDC-lite SSO 허브. `lib/sso*`, `app/sso/*`, `app/.well-known/jwks.json`, 마이그레이션 M010·M011, 로그인 페이지 `next` 새니타이즈 포함.

- 상태: **구현 완료(로컬 게이트 통과) / Vercel env 세팅·스테이징 E2E 대기**

## 1. 배경 / 문제

사내에 동일 임직원(`@eland.co.kr`, ~1,800명)을 대상으로 하는 Next.js 앱이 다수 존재하나(AI캠퍼스, web/fashion, measure-web, OPR 등) **앱마다 독립 로그인**이라 동일인이 여러 번 로그인해야 한다.

모든 앱이 `@eland.co.kr` 도메인 + Vercel(`*.vercel.app`) 환경을 공유하지만, `*.vercel.app`은 Public Suffix List(PSL)에 등재되어 있어 브라우저가 `domain=.vercel.app` 쿠키 설정을 **구조적으로 거부**한다. 쿠키 공유형 SSO가 불가능하므로 **리다이렉트 URL로 단명 토큰을 전달하는 토큰 핸드오프 방식**을 채택한다.

AI캠퍼스는 이미 가장 완성된 회원·OTP·역할 체계(master/admin/user, bcrypt, JWT httpOnly, 감사 로그)를 보유하므로 **중앙 IdP로 승격**하면 한 번 로그인으로 전 앱에 접속할 수 있다.

## 2. 목표 / 비목표

### 목표

- G1. AI캠퍼스에 OIDC-lite SSO 엔드포인트(`/sso/authorize`, `/.well-known/jwks.json`, `/sso/logout`, `/sso/userinfo`)를 신설한다.
- G2. 허브가 발급하는 **단명(60초) id_token**으로 스포크에 신원(email)을 전달한다.
- G3. 토큰 서명은 **RS256 + 공개 JWKS**로 하여 스포크에 개인키를 확산하지 않는다.
- G4. 스포크는 받은 email로 자기 DB에서 provision/lookup 후 자기 세션 쿠키를 자기 시크릿으로 발급한다(인가·역할은 스포크가 재조회).
- G5. 추가 비용 0 — Vercel Hobby / Neon Free / 기존 `jose` 의존성 / 신규 유료 서비스 금지.
- G6. 기존 각 앱 자체 로그인을 유지(하이브리드)하며 점진 전환. 기존 HS256 `user_session`과 공존.

### 비목표

- N1. 표준 OAuth2/OIDC 완전 준수 아님 — authorization_code+PKCE, refresh_token, discovery document 전체는 v1 범위 밖.
- N2. 인가(권한/역할) 중앙화 아님 — 허브 토큰은 "누구인지"만 전달. 역할은 각 스포크가 자기 DB에서 재조회.
- N3. 글로벌 로그아웃(전체 앱 동시) 아님 — 각 앱은 자기 세션만 독립 로그아웃.
- N4. Capacitor/WebView 딥링크 SSO — v1은 웹 리다이렉트만.
- N5. SCIM 프로비저닝·사용자 디렉터리 통합 아님.

## 3. 사용자 시나리오

### S1. 스포크 앱(예: measure-web) 첫 방문 — 허브 미로그인

1. 사용자가 measure-web 보호 페이지 접근 → 세션 없음 → 스포크가 `state`/`nonce`/`returnTo` 생성·저장.
2. 스포크 → `https://retail-ai-campus.vercel.app/sso/authorize?app=measure-web&redirect_uri=https://<measure>/sso/callback&state=<csrf>&nonce=<n>` 로 302.
3. 허브 `/sso/authorize`: 레지스트리(`sso_clients`) 검증 → 허브 세션 없음 → `/login?next=/sso/authorize?...`로 302.
4. 사용자 허브 로그인 완료 → `next` 파라미터로 `/sso/authorize?...` 복귀.
5. 허브: id_token 발급(RS256, sub=email, aud=measure-web, exp=iat+60) + nonce 저장(`sso_nonces`) → `https://<measure>/sso/callback?token=<id_token>&state=<csrf>` 302.
6. measure-web `/sso/callback`: state 검증 → JWKS RS256 검증 → email provision/lookup → `measure_session` 쿠키 발급 → `returnTo`로 302.

### S2. 스포크 방문 — 허브 이미 로그인 상태

1. 스포크 → `/sso/authorize?...` 302.
2. 허브: 세션 있음 → 로그인 화면 없이 즉시 id_token 발급 → 스포크 콜백 302.

### S3. 로그아웃

1. 스포크에서 자기 세션 쿠키만 삭제 → 자기 로그인 화면으로 이동. 다른 앱 세션은 각자 TTL까지 유지.
2. 허브에서 로그아웃하면 `user_session` 쿠키 삭제 → 허브 홈(또는 `post_logout_redirect_uri`).

## 4. 기능 요구사항

### F1. 구현된 파일 목록

| 파일 | 역할 |
|---|---|
| `lib/sso-keys.ts` | `importPKCS8`/`importSPKI`로 RS256 키 로드, `exportJWK`로 공개 JWK 생성 |
| `lib/sso-clients.ts` | `sso_clients` DB 조회(우선) + `SSO_CLIENTS_JSON` env 폴백, 정확매칭 화이트리스트 |
| `lib/sso-nonce.ts` | `sso_nonces` 테이블 INSERT(발급)·consumed UPDATE(1회성 소비) |
| `lib/sso.ts` | `signIdToken` — RS256 SignJWT, TTL 60초, 클레임 `{ iss, sub, aud, nonce, iat, exp }` |
| `app/sso/authorize/route.ts` | GET — 레지스트리 검증·세션 확인·id_token 발급·302 |
| `app/sso/logout/route.ts` | GET — `user_session` 삭제·`post_logout_redirect_uri` 화이트리스트 검증·302 |
| `app/sso/userinfo/route.ts` | GET — `Authorization: Bearer` 검증 후 최소 프로필 반환(email/name 등) |
| `app/.well-known/jwks.json/route.ts` | GET — RS256 공개 JWK Set 반환, `Cache-Control: public, max-age=600` |
| `app/login/page.tsx` | `next` 쿼리 파라미터 지원·동일 오리진 새니타이즈·모바일 WebView 감지 추가 |

### F2. 엔드포인트 명세

**`GET /sso/authorize`**

쿼리 파라미터: `app`(필수), `redirect_uri`(필수, 정확매칭), `state`(필수), `nonce`(선택), `prompt`(선택: `none`).

처리 순서:
1. `app` 조회 → 없으면 400(리다이렉트 없음).
2. `redirect_uri` 정확매칭 실패 → 400(리다이렉트 없음). 오픈리다이렉트 1차 방어.
3. `state` 누락 → 400.
4. 허브 세션 확인: 없음 + `prompt=none` → `error=login_required&state=` 리다이렉트. 없음(일반) → `/login?next=<urlencoded authorize URL>`.
5. 세션 있음 → `nonce` 생성(또는 스포크 제공값 사용) → `sso_nonces`에 저장 → id_token 발급 → `redirect_uri?token=&state=` 302.

**`GET /.well-known/jwks.json`**

응답: `{ "keys": [{ kty, use, alg:"RS256", kid, n, e }] }`. 개인키 미포함. `Cache-Control: public, max-age=600`.

**`GET /sso/logout`**

`user_session` 쿠키 삭제. `post_logout_redirect_uri` 있으면 화이트리스트 검증 후 302. 다른 앱 세션 미변경.

**`GET /sso/userinfo`**

`Authorization: Bearer <id_token>` 검증(RS256, iss/aud/exp). 통과 시 `{ email, name, corporation_name, organization_name, position }` 반환. 역할/권한 미포함.

### F3. 토큰 스펙

- 서명: RS256(`importPKCS8`, `SignJWT` from `jose`)
- TTL: **60초** (exp = iat + 60)
- 헤더: `{ alg: 'RS256', kid: <SSO_KID>, typ: 'JWT' }`
- 클레임: `iss=SSO_ISSUER`, `sub=user.email(lowercase)`, `aud=app`, `nonce`, `iat`, `exp`
- 역할/권한 클레임 없음(인가는 스포크 책임)

### F4. 클라이언트 레지스트리 (`sso_clients`)

DB 테이블 우선, `SSO_CLIENTS_JSON` env 폴백. `redirect_uri` 정확매칭만 허용(prefix/와일드카드 금지). `https`만 허용(운영). 마이그레이션 M010으로 테이블 생성.

### F5. nonce 1회성 (`sso_nonces`)

마이그레이션 M011으로 테이블 생성. 발급 시 `INSERT`, 소비 시 `UPDATE consumed=true WHERE consumed=false`. 재사용·만료(발급+90초) 시 거부. Neon 테이블 사용(Upstash Redis 미사용).

### F6. 마이그레이션

| ID | 대상 | 패턴 |
|---|---|---|
| M010 | `sso_clients` 테이블 생성 | `CREATE TABLE IF NOT EXISTS`, `requireMaster` 보호 |
| M011 | `sso_nonces` 테이블 생성 | `CREATE TABLE IF NOT EXISTS`, `requireMaster` 보호 |

`app/api/admin/migrate/route.ts` 기존 패턴(멱등, 마스터 전용) 그대로 확장. `supabase/schema.sql` 동기화 완료.

### F7. 로그인 페이지 `next` 파라미터

- `next`는 `/`로 시작하는 동일 오리진 경로만 허용(오픈리다이렉트 방지).
- 모바일 WebView(Capacitor) 감지 시 SSO 리다이렉트 대신 앱 내부 로그인 처리.
- 로그인 성공 후 `next`가 유효하면 해당 경로로 302.

### F8. 런타임

모든 신설 SSO 라우트에 `export const runtime = 'nodejs'` 명시. Vercel Hobby에서 Edge 런타임은 Node `crypto`/RS256 키 임포트 제약으로 금지.

## 5. UX / 디자인

- **사용자 흐름**: 스포크에서 "AI캠퍼스로 로그인" 버튼 클릭 → 허브 로그인 화면(미인증 시) → 인증 완료 후 자동으로 원래 스포크 페이지로 복귀. 사용자는 토큰·리다이렉트 내부 흐름을 인지하지 않는다.
- **기존 로그인 공존**: 스포크 기존 자체 로그인(이메일+비번)은 그대로 유지(하이브리드). SSO는 "세션을 얻는 또 다른 입구"일 뿐.
- **로그인 페이지 UX**: `next` 파라미터가 있어도 UI 변화 없음. 로그인 성공 후 자동 복귀.
- **에러 UX**: `redirect_uri` 불일치·`app` 미등록은 허브에서 직접 400 응답(리다이렉트 없음). `error=login_required`는 스포크가 안내 처리.
- **JWKS 캐시**: `max-age=600`(10분). 스포크 `createRemoteJWKSet`이 내부적으로 캐싱 처리하므로 스포크 작업자가 별도 캐시 구현 불필요.

## 6. 엣지 케이스

| ID | 상황 | 처리 |
|---|---|---|
| E1 | `redirect_uri`가 레지스트리에 없음 | 400 반환. 리다이렉트 자체 금지. 토큰 미발급. |
| E2 | `state` 누락 | 400. |
| E3 | `prompt=none`이고 허브 세션 없음 | `error=login_required&state=<s>` 로 조용히 리다이렉트(silent 체크). |
| E4 | `next` 파라미터가 외부 URL | 동일 오리진 새니타이즈로 거부(오픈리다이렉트 차단). 기본값(`/`)으로 대체. |
| E5 | nonce 재사용(replay) | `consumed=true` 이므로 UPDATE가 0 rows → 거부. |
| E6 | 60초 경과 토큰 | `jose` `jwtVerify`의 `exp` 자동 검증으로 거부(스포크 측). |
| E7 | `aud` 불일치 토큰 | 스포크 검증 시 `audience` 옵션으로 거부. |
| E8 | JWKS 키 회전 직후 스포크 캐시 불일치 | 신·구 공개키 동시 노출(겹침 기간). `createRemoteJWKSet` kid 매칭으로 자동 해소. |
| E9 | `@eland.co.kr` 외 이메일로 허브 통과(예외 허용 계정) | 스포크가 자기 allowlist 재검증 후 provision 거부. |
| E10 | measure-web·OPR 운영 도메인 미확정 | `sso_clients` 미등록 상태 → 해당 앱 SSO 동작 불가. URL 확정 후 DB 행 추가. |
| E11 | 허브 로그아웃 후 스포크 세션 잔존 | host-only 쿠키라 허브가 삭제 불가(best-effort). 각 스포크 TTL까지 잔존(비목표 N3 per 확정). |
| E12 | Capacitor WebView에서 SSO 리다이렉트 | 모바일 감지 후 앱 내부 로그인 처리. WebView 딥링크 SSO는 v1 범위 밖(N4). |
| E13 | `SSO_PRIVATE_KEY` env 미설정 | `lib/sso-keys.ts` 오류 → `/sso/authorize` 500. Vercel env 세팅 선행 필수. |

### §6 보안 섹션 — CLAUDE.md §6 정책 매핑

| CLAUDE.md §6 | SSO 적용 |
|---|---|
| §6-1 `@eland.co.kr` 도메인 | 허브 `user_session`(기존 OTP 인증 통과자만) + 스포크 provision 시 재검증 |
| §6-4 JWT httpOnly | `user_session`(허브 기존 쿠키) 무변경. id_token은 URL 경유 후 즉시 소비(60초). |
| §6-5 레이트리밋 | `/sso/authorize`에 기존 `lib/ratelimit.ts`(Upstash) IP 단위 적용 |
| §6-7 PII 보호 | id_token에 email만. `userinfo`도 최소 프로필. 로그에 토큰 원문 미기록. |
| §6-8 에러 통일 | catch → "서버 오류가 발생했습니다." raw 노출 금지 |
| §6-10 감사 로그 | SSO 발급/거부 이벤트 `auth_logs` 기록 |

**PII 보호**: id_token은 email(sub)만 포함. `userinfo`도 역할/권한 미포함. 서버 로그에 토큰 원문·email 평문 미기록.

**개인키 비노출**: `SSO_PRIVATE_KEY`는 허브 Vercel env에만. 스포크·소스 코드·클라이언트 미노출. `/.well-known/jwks.json`은 공개키만 반환.

**DB 마이그레이션 안전성**: M010·M011 모두 `CREATE TABLE IF NOT EXISTS`(멱등). 기존 테이블 미변경. 하위호환.

**env 동기화**: `.env.local.example`에 `SSO_PRIVATE_KEY`, `SSO_PUBLIC_KEY`, `SSO_KID`, `SSO_ISSUER`, `SSO_CLIENTS_JSON`(선택) 추가 완료. `.gitleaks.toml` allowlist에 자리표시자 등록.

## 8. 미해결 질문

1. **Vercel env 세팅(차단 항목)**: `SSO_PRIVATE_KEY`·`SSO_PUBLIC_KEY`·`SSO_KID`·`SSO_ISSUER` — Vercel 대시보드에 멀티라인 PEM 등록 필요. 이 작업 전까지 AC2/3/4/7/8 스테이징 검증 불가.
2. **스포크 운영 URL 확정(Q6)**: measure-web·OPR 실제 배포 도메인이 미확정. 확정되는 대로 `sso_clients`에 행 추가(`https` only, 정확매칭). web-fashion(`eland-apparel.vercel.app`)은 코드상 확인이나 사용자 최종 확인 대기.
3. **measure-web 파일럿 착수 시점**: 스포크 운영 URL 확정 + `sso_clients` 등록 후. 스포크 레포에 `/sso/callback` 구현 필요(계약은 `docs/sso-spoke-integration-contract.md` 참조).
4. **`@eland.co.kr` 예외 계정과 스포크 allowlist 불일치 처리**: 허브 allowlist 예외 통과 계정이 스포크 allowlist에는 없을 때의 정책 합의 필요.
5. **향후 강화 경로(문서화)**: URL 토큰 노출 최소화 시 fragment(`#token=`) 또는 form_post 방식. 보안 요구 상승 시 `code`+`POST /sso/token` 코드 교환 방식으로 승격 가능(현재 인터페이스 확장 설계).
